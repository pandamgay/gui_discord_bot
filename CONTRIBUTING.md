# Branch 및 PR 규칙

## Branch 전략 및 규칙

본 프로젝트는 git-flow 전략 및 규칙에 따릅니다.

- **main**: 프로젝트가 배포되는 브랜치
- **develop**: 다음 배포를 위한 개발하는 브랜치
- **feature**: develop 브랜치에 포함되며, 추가 기능을 개발하는 브랜치
- **release**: 다음 버전 출시를 준비하는 브랜치
  - **feature**: 브랜치에서 release 브랜치로 QA 및 테스트 진행 후 main 브랜치로 머지
- **hotfix**: main 브랜치의 버그를 수정하는 브랜치

> 브랜치는 접두사 규칙을 지켜야 하며, develop/main에서 분기

## PR(Pull Request) 규칙

- PR 제목: `[브랜치종류] 간단한 설명`
 - 예: `[feature] 로그인 화면 추가`

- PR 내용
  1. 변경 목적
  2. 주요 변경 사항
  3. 확인 방법
  4. 참고 사항 (필요 시)
- 리뷰 전 체크리스트

 - [ ] 기능 동작 확인
 - [ ] 코드 스타일 일관성 확인

## 커밋 메시지 규칙

### 커밋 제목

- 커밋 제목: `타입: 주요 수정사항`
 - 예: UPDATE: 로그인 API구현

- 타입 목록
  - [INITIAL] — repository를 생성하고 최초에 파일을 업로드 할 때
  - [ADD] — 신규 파일 추가
  - [UPDATE] — 코드 변경이 일어날때
  - [REFACTOR] — 코드를 리팩토링 했을때
  - [FIX] — 잘못된 링크 정보 변경, 필요한 모듈 추가 및 삭제
  - [REMOVE] — 파일 제거
  - [STYLE] — 디자인 관련 변경사항

### 커밋 메시지

- 커밋 메시지: `자세한 수정사항`
