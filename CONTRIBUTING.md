# Branch 및 PR 규칙

## Branch 전략 및 규칙

본 프로젝트의 전략 및 규칙은 다음과 같습니다.

- **main**: 프로젝트가 안정화된 상태로 릴리즈 배포에 진입합니다.
- **develop**: 다음 배포를 위한 개발하는 브랜치
- **release**: 프로젝트 릴리즈가 배포되는 브랜치

### Branch flow
develop → main → release 순으로 병합됩니다.

### Tag 및 Release 노트 관리
태그는 release 브랜치 기준으로 생성합니다.  
릴리즈 노트는 태그 생성 시 함께 작성합니다.

### Hotfix 정책
release에서 버그 발견 시 develop에서 수정 후 release에 반영합니다.  
긴급 수정 필요 시 별도 hotfix/* 브랜치 생성 가능합니다.  
release에 반영된 수정은 develop에도 동일하게 반영합니다.

## PR(Pull Request) 규칙

- PR 제목: `[브랜치종류] 간단한 설명`
 - 예: `[feature] 로그인 화면 추가`

- PR 내용
  1. 변경 목적
  2. 주요 변경 사항
  3. 확인 방법
  4. 참고 사항 (필요 시)
- 리뷰 전 체크리스트

 - [ ] 기능 동작 확인
 - [ ] 코드 스타일 일관성 확인

## 커밋 메시지 규칙

### 커밋 제목

- 커밋 제목: `타입: 주요 수정사항`
 - 예: UPDATE: 로그인 API구현

- 타입 목록
  - [INITIAL] — repository를 생성하고 최초에 파일을 업로드 할 때
  - [ADD] — 신규 파일 추가
  - [UPDATE] — 코드 변경이 일어날때
  - [REFACTOR] — 코드를 리팩토링 했을때
  - [FIX] — 잘못된 링크 정보 변경, 필요한 모듈 추가 및 삭제
  - [REMOVE] — 파일 제거
  - [STYLE] — 디자인 관련 변경사항

### 커밋 메시지

- 커밋 메시지: `자세한 수정사항`
